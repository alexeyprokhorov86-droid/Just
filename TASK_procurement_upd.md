# TASK_procurement_upd — Автоматизация приёмки УПД и создания ПТУ в 1С

**Старт**: 2026-04-22. **Владелец**: Алексей. **Оценка**: MVP 3-4 недели.

## Пользовательский сценарий (as described)

Кладовщик в TG (позже Element X):
1. Вызывает команду `/receive` → бот просит фото УПД.
2. Загружает 1+ фото УПД, подтверждает «это всё».
3. Бот распознаёт УПД (Claude Vision), показывает кладовщику что понял.
4. Бот находит кандидатов Заказа поставщику по критериям → кладовщик выбирает (если >1).
5. Бот проверяет Соглашение; если >1 действующих — спрашивает в группе Закупки, какое одно реально.
6. Бот просит скан этикетки товара (для серий, если вид номенклатуры требует серий).
7. Бот создаёт ПТУ в 1С → заполняет поля из Заказа/Соглашения/УПД/прочего → прикрепляет сканы → проводит.
8. Бот репортит результат в группу Закупки: успех/неуспех. При НДС — напоминает про счёт-фактуру. Мониторит списание БД-средств (5 мин).

## Критерии кандидата-Заказа поставщику

- проведён (`posted=true`)
- статус `Подтверждён`
- не старше 3 месяцев
- поставщик НЕ в чёрном списке (надо понять где этот список в 1С)
- номенклатура НЕ в группе `Архив` (проверка по иерархии `Catalog_ВидыНоменклатуры`)
- по нему нет приобретений ИЛИ сумма приобретений < сумма заказа

## Риски

1. **OData write недоступен / ограничен** — например, нельзя создавать ПТУ через OData (только через Web-форму). Митигация: Фаза 0.1 делает smoke-test на минимальном документе; если не работает — эскалация до старта Фазы 1.
2. **Качество OCR на мятых сканах / рукописных пометках** — проверяем на 2-3 образцах в Фазе 0.5. Если <80% полей верно, нужен fallback (ручная корректировка перед созданием ПТУ).
3. **Временная рассинхронизация `c1_*`** — бот создаёт ПТУ в 1С, но в нашей БД он появится только после `sync_1c_full --incremental`. Решение: после `Post` сразу дёргаем `sync_1c_full --incremental` для `c1_purchases` (или обновляем точечно).
4. **Соглашение "действующее" — нечёткое понятие** — в 1С может быть не один флаг, а комбинация (срок × дата документа × приоритет). Уточнить с Алексеем в Фазе 0.3.
5. **Тест-база vs прод** — Алексей сказал «будем тестить сначала». Детали: есть ли отдельная база, или это прод, но с соглашением аккуратности? Уточнить.

### Новые правила валидатора УПД (из обсуждения 2026-04-22)

1. **Перевёрнутые роли supplier/buyer** = **невалидный документ**. Алерт в группу Закупки с вариантами «принять с ручной правкой / отклонить / уточнить у поставщика». **Блокируем создание ПТУ** до ответа из группы.
2. **Ставка НДС** — допустимы `{0, 10, 20, 22, "Без НДС"}`. Если что-то другое — алерт.
