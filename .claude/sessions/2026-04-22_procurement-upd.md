4. **Баги/квирки OData 1С** (все обойдены в коде):
   - `$filter` с русскими значениями (`Статус eq 'Подтвержден'`) → 500. Фильтруем в Python.
   - `$filter=DeletionMark eq false` ломает выдачу у `Catalog_ВидыНоменклатуры`, `Catalog_СерииНоменклатуры`, `Catalog_Контрагенты`. Фильтруем в Python.
   - Имена полей кириллицей в $filter (`Партнер_Key`) → 500. Фильтруем в Python.
   - `requests.params` кодирует `:` в `%3A` в datetime → 500. Собираем URL вручную (как в `sync_1c_full.py`).
   - `$top=500` + `$orderby=Date desc` иногда 500. Использовать `$top=100` + пагинация.
   - Даты `0001-01-01` в 1С = «пусто/бесконечность», не «уже истекло».

5. **Claude Vision галлюцинирует имена, но не ИНН**:

### Фаза 3 — Создание ПТУ в 1С через OData

Самая рискованная фаза: **первая реальная запись в прод-базу 1С**. Нужна осторожность.

1. **Перед кодом** — протестировать создание документа минимальным POST на тест-данных. Идея:
   - Скопировать payload **существующего** проведённого ПТУ через GET
   - Убрать `Ref_Key`, изменить `Number`, может `Date`
   - POST → проверить что создался новый
   - Post → убедиться что провёлся
   - Unpost + DELETE → откат, чтобы не засорять прод.

2. **`tools/onec_write.py`** — низкоуровневый слой:
   - `create_document(doc_type, payload) -> ref_key`
   - `post_document(doc_type, ref_key)`
   - `unpost_and_delete(doc_type, ref_key)` — для rollback
   - `create_and_post(doc_type, payload)` — композитный с автоматическим rollback при ошибке Post.

3. **`tools/procurement_builder.py`** — сборка payload ПТУ из:
   - Выбранного Заказа поставщику (все 80 полей в raw)
   - Распознанного УПД (номер/дата/наименование входящего, позиции)
   - Действующего Соглашения (валюта, оплата, склад, менеджер и т.д.)
   - Пользователя (`identification.identify_employee_by_text` → `Принял_Key`, `ПринялДолжность`)
   - Организации (руководитель, гл. бухгалтер)
   - Контрагент счёта (из `Catalog_КонтрагентыБанковскиеСчета` — пока не синхронизирован, надо добавить)
   - Товары (из Заказа с lookup Серия_Key — но Фаза 4 добавляет серии, пока без них)

4. **Проверка Соглашения**:
   - `SELECT * FROM c1_supplier_agreements WHERE partner_key=... AND organization_key=... AND approved AND (valid_from<=now() AND (valid_to>=now() OR valid_to='0001-01-01'))`
   - 0 результатов → блокер, alert в Закупки
   - 1 → используем
   - >1 → вопрос в Закупки с кнопками (требует отдельного State + persistent store для ожидания ответа)

5. **Integration в receive_flow.py**:
   - Новое state `CONFIRMING` после `CHOOSING_ORDER`
   - Кнопка «Создать ПТУ» → сборка payload → preview → «Подтвердить» → POST → Post → показ номера нового ПТУ
   - Ошибки POST → показ + кнопка «Повторить» / «Отмена»

### Риски Фазы 3

- **Обязательные поля** — из 95 полей ПТУ часть nullable, часть нет. Метаданные не всегда точно отражают валидацию в 1С. Первый POST может упасть на unknown required field — нормально, отлаживать итеративно.
- **Табличная часть `Товары`** — формат в OData: передавать как массив словарей с теми же ключами. Без `Ref_Key` в строках (1С генерирует).
- **`PostingModeOperational`** — при Post. Для даты в прошлом = false, для сегодня = true. Текущая документации нет, надо тестить.
- **Rollback**: если POST создал документ, но Post упал — документ висит непроведённый. Обязательно `Unpost + DELETE` в rollback. Идея: перед Post сохранять `ref_key` в отдельный файл `pending_ptu.log` — при падении скрипта сможем восстановить и вычистить.
